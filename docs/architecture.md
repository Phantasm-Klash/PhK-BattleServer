# PhK-BattleServer Architecture

Status: v0.1 skeleton.

## Runtime Role

`PhK-BattleServer` is the open-source C++ authority for high-frequency PVP and Boss battle simulation. Gensoulkyo allocates matches and signs short-lived battle tickets; SpellKard connects to the allocated battle endpoint and sends only input or intent packets.

## Module Layout

| Module | Responsibility |
| --- | --- |
| `phk/v1/manifest.hpp` | Generated dependency-light protocol manifest from `PhK-Protocol`. It currently supplies shared version/ruleset constants and message-field gates until full protobuf C++ bindings replace it. The repository checker also reads the exported JSON descriptor directly to pin key enum values and field number/type shapes for the protobuf replacement path. |
| `ticket` | Holds the battle ticket shape and development verifier. The current verifier checks Ed25519 key/signature shape and ticket binding, but does not perform production crypto yet. |
| `handshake` | Holds the ECDHE/AEAD handshake boundary. The current implementation checks hello key/random/AEAD shape, selects ChaCha20-Poly1305-compatible labels, derives deterministic development session/key references, and emits Ed25519-shaped transcript signature material. |
| `kcp_endpoint` | Holds the KCP/UDP endpoint boundary. The current implementation is an echo placeholder plus a server-backed encrypted packet adapter that gates AEAD/session/tick/reconnect checks before forwarding accepted datagrams to the placeholder endpoint. |
| `protocol` | Holds battle packet headers, encrypted packet adapter structs, and dispatcher guards until generated protobuf bindings are wired. It rejects client-authored result packets and now requires development key id/nonce/ciphertext/auth-tag shape, header-bound development nonce derivation, and nonce reuse rejection on encrypted-path packet types before accepting seq state. |
| `simulation` | Holds the v0.1 deterministic battle-core slice: fixed 60Hz tick, match-bound mode/ruleset/seed metadata, authoritative input and mode-action validation, shared mode-action type gating, input seq/tick windows, tick-bound mode-action buffering/application including reconnect intent, missing-input fallback audit counters, deterministic replay input/event trace material, manifest-compatible replay-summary record canonical/hash material that includes the match seed, canonical replay fixture payload/hash material that binds the match seed plus final snapshot players/bullets/mode state, replay-record/loadout bridge canonical/hash material for the future `ReplayRecord` protobuf export, replay-bound development mode-result JSON material, player disconnect/reconnect state, cursor-aware reconnect snapshots, milli-unit movement, simplified bullet generation/movement, canonical state hash, replay summary hashes, replay fixture material, and lightweight mode-action audit projection. |
| `result` | Holds the battle result shape and development verifier. It checks match/mode binding, player ids, match seed binding, result hash, replay id, replay event cursor, replay input/event/final-state digest binding, projection-only reward shape, settled time, Ed25519 signature field shape, and deterministic development signature binding to the canonical result payload before Gensoulkyo accepts a battle result. |
| `server` | Composes ticket verification, capacity-checked session creation, registered-ticket handshake acceptance, negotiated-session encrypted packet validation, handshake-bound encrypted packet dispatch, a decoded packet adapter plus decoded header/payload/session binding for future protobuf input, mode-action, and reconnect handoff, match simulation input/tick/snapshot/reconnect calls, replay/seed-bound development signed-result callback material, ticket-derived replay loadout export material, and idempotent battle result submission bound to the local replay summary. |

## Planned Production Replacements

1. Replace structural ticket checks with real Ed25519 verification.
2. Replace handshake placeholder with X25519 ECDHE, HKDF, transcript hashing, and server signatures.
3. Replace KCP echo placeholder with a real UDP/KCP event loop.
4. Replace the dependency-light C++ manifest plus local packet structs with generated `PhK-Protocol` C++ protobuf bindings.
5. Replace the structural battle result verifier with real Ed25519 signing/verification and submit the result to Gensoulkyo over the service-to-service channel.
6. Expand the current fixed-tick simulation from simplified movement/bullet snapshots into full card, Boss, hit/graze, scoring, snapshot/event encoding, replay input stream digests, and signed result submission back to Gensoulkyo.

## Security Rules

- A battle ticket is bound to `match_id`, `player_id`, `mode_id`, `battle_server_id`, endpoint, deck snapshot hash, ruleset version, nonce, and expiry.
- A match simulation freezes the first registered ticket's `mode_id` and `ruleset_version`; later tickets for the same match must match both.
- Match session count cannot exceed the configured battle-server `max_players`; Boss modes additionally clamp join capacity to the protocol-planned 4-8 player range even if a local server config is accidentally wider.
- Reused ticket ids are rejected by the server facade.
- Handshake acceptance re-verifies the ticket structure/expiry and requires the ticket to be registered in the server facade.
- Handshake hello messages must provide non-empty client key/random material and at least one supported ChaCha20-Poly1305-compatible AEAD label.
- Handshake accepts include deterministic development client-to-server and server-to-client key references plus an Ed25519-shaped server signature over the transcript hash. These are boundary fixtures for future X25519/HKDF/transcript signing and do not replace production cryptography.
- Encrypted-path packet dispatch is not enabled by ticket registration alone. A registered session must first complete the handshake; after acceptance, the facade replaces the temporary ticket signing key with the handshake-derived client-to-server key reference for inbound packet `key_id` checks, while retaining the server-to-client key reference as outbound session material.
- Battle input and mode actions must come from a player with a registered server session for that match.
- Battle input and mode-action seq numbers must increase within the configured server window; old, replayed, or implausibly jumped seq values are rejected.
- A player may queue only one input for a given authoritative tick. A later higher-seq packet for the same player/tick is rejected instead of replacing the buffered input, so replay hash material and applied simulation state cannot diverge.
- Mode actions are limited to the shared server-interface action surface: `cast_card`, `select_round_card`, `transfer_card`, `ready`, and `reconnect`. Unknown action types are rejected until full protobuf/native mode dispatch replaces this scaffold.
- Accepted mode actions are buffered by target tick and only folded into replay/event/hash state when the authoritative simulation advances that tick.
- Disconnected players remain blocked from input and normal mode actions, but `reconnect` mode actions are accepted as the narrow recovery intent and restore connected state only when their target authoritative tick is processed.
- Boss mode `boss_start_ready` is derived from connected player count, so a registered 4-player Boss room is not start-ready while one player is disconnected.
- Boss modes may allocate and wait with fewer than 4 registered players, but the server facade refuses to build signed battle results, replay records, or accept submitted results for an underfilled Boss match. This keeps the lobby/waiting lifecycle separate from settlement authority and enforces the documented 4-8 player Boss start boundary.
- Boss signed-result projection also binds `boss_min_players`, `boss_max_players`, `boss_registered_player_count`, `boss_start_ready`, `boss_ready_player_count`, `boss_all_registered_connected`, `boss_all_registered_ready`, `boss_ready_to_start`, `connected_player_count`, `disconnected_player_count`, and each player's server-generated spawn slot/fire target back to the final replay snapshot, so Nakama/Go can audit the C++ start-readiness, connection state, and layout without trusting client-authored lobby fields.
- Boss snapshots expose each player's server-generated ring spawn slot and fixed `boss_center` fire target in `mode_state`, so world/instance Boss instance layouts are visible to replay/result fixture hashing without letting clients author positions or damage.
- Boss `transfer_card` mode actions must reference a server-configured in-match transferable card state before they can enter the authoritative tick queue. The simulation rejects unknown card instances, owner mismatches, mode-forbidden cards, unpaid costs, and blocked cooldowns before seq/replay/hash state advances. Accepted transfers freeze the server authority state into snapshot mode state, replay trace/hash material, and development mode-result projection for later business-server audit. This is an in-memory battle authority bridge for the future card/rules config path, not inventory, reward, wallet, or database ownership.
- Client packets cannot submit battle results.
- Disconnected players cannot submit battle input or mode actions until the server facade marks them connected again.
- Encrypted-path input and mode-action packets from disconnected players are rejected at the server facade before dispatcher seq/nonce state advances.
- Reconnect snapshots are full authoritative snapshots that include the server event cursor, requested cursor, and missed event count; a client cursor ahead of the server cursor is rejected as `event_cursor_ahead`.
- The local `BattlePayloadType` enum is pinned to the v0.1 protobuf numeric order in CTest, and the checker validates the exported descriptor's `BattlePayloadType` enum plus key byte-field number/type shapes. This is a drift gate for generated protobuf replacement, not a runtime decoder.
- Encrypted-path battle packets must carry a key id, a development nonce derived from direction, key id, match/player identity, payload type, tick, seq, and ack, non-empty ciphertext, 16-byte AEAD tag, and an input/mode-action/ping/reconnect payload type in the dispatcher scaffold before seq replay state advances. A hex-shaped nonce that does not match the header-bound development derivation is rejected, and reusing the same `match_id:player_id:key_id:nonce` on the encrypted adapter path is rejected even if the seq is new. This remains a deterministic adapter contract for future ChaCha20-Poly1305 nonce construction, not production AEAD verification.
- Server-facade encrypted packet dispatch additionally requires the packet to be a client-to-server encrypted payload type (`input`, `mode_action`, `ping`, or `reconnect`), and the packet's `match_id`/`player_id` to resolve to a registered battle session that has completed handshake with a matching inbound handshake key reference. Snapshot/event/handshake/result payloads are rejected at the facade before dispatcher seq/nonce state advances. Input, mode-action, and ping packet acknowledgements cannot claim a snapshot tick beyond the authoritative simulation tick, and input/mode-action packets must also stay inside the authoritative simulation tick window before dispatcher seq/nonce state changes. Reconnect packet acknowledgements cannot claim an event cursor beyond the server replay summary. Unknown sessions, missing handshake, mismatched key ids, stale ticks, far-future input ticks, impossible ping/input acknowledgements, impossible reconnect cursors, or server-to-client payload attempts are rejected at the facade.
- The facade's negotiated-session validation also requires handshake transcript material, a supported ChaCha20-Poly1305-compatible selected AEAD label, non-empty client-to-server and server-to-client key refs, and distinct directional key refs before encrypted packet dispatch reaches dispatcher seq/nonce state. Inbound packets using the outbound server-to-client key ref are rejected as session key mismatches. These remain deterministic development guards for the future real X25519/HKDF/AEAD session object.
- Decoded protobuf-replacement input and mode-action handoff is bound to the already-validated packet header and negotiated session before simulation state changes. The decoded local `BattleInput` or `BattleModeAction` must match header version stamp, `match_id`, `player_id`, `tick`, `seq`, and payload type; the direct decoded facade also requires a registered session that completed handshake, a matching client-to-server key reference, and a snapshot `ack` that is not ahead of the authoritative simulation tick. Mismatches are rejected as header/payload or session-boundary failures and do not advance replay counters or queued mode-action state.
- The development decoded packet adapter models the future AEAD decrypt + protobuf decode boundary as a two-step handoff: encrypted packet/session/nonce/tick checks must pass first, then exactly one decoded input, mode-action, or reconnect intent payload must match the header. `DecodedBattlePacketResult.encrypted_dispatch_accepted` is only set after that encrypted dispatch precondition succeeds, so missing handshake/session-key/tick failures are visibly separate from decoded-payload rejections. Missing decoded payloads, decoded-kind mismatches, pings with decoded battle payloads, or failed encrypted dispatches do not advance simulation replay counters or queued mode-action state.
- Decoded reconnect handoff is intentionally narrower than generic mode actions: a reconnect packet must decode to a `reconnect` mode action bound to the encrypted header's version, match id, player id, tick, seq, and `last_seen_event_cursor` before it can recover transport state. The decoded cursor must equal the encrypted header `ack` and cannot be ahead of the server replay event cursor; accepted decoded reconnects mark the server facade player connection active immediately while the reconnect mode action remains queued for deterministic authoritative tick/replay recording.
- The development KCP/AEAD adapter calls the server encrypted dispatcher before handing a datagram to the KCP placeholder. Rejected encrypted packets do not increment KCP endpoint stats or produce replies; accepted packets are the only path forwarded to the placeholder endpoint. This is a testable replacement point for the future real KCP event loop and AEAD decrypt/decode path.
- The development KCP/AEAD adapter also rejects UDP frames with missing remote endpoint identity or empty datagram payload before encrypted dispatch, so malformed transport frames cannot consume seq/nonce state or reach the placeholder KCP endpoint while the real event loop is pending.
- Remote endpoint rebinding is limited to reconnect packets for sessions currently marked disconnected; connected players cannot switch UDP endpoints with a reconnect-shaped packet. This keeps NAT/reconnect recovery explicit while the real KCP event loop is pending.
- Development signed-result callback material is generated only from the server-owned replay summary, manifest-compatible replay summary record, and canonical replay fixture: allocated match id, frozen mode id/ruleset, match seed, player ids, replay-derived result hash, replay id, event cursor, final tick, fixed tick rate, missing-input fallback counters, accepted mode-action count, deterministic input/event trace counts, input stream hash, event stream hash, final state hash, replay summary record hash, canonical replay fixture hash, projection-only reward JSON, battle server key id, and deterministic development signature material bound to the canonical result payload.
- Boss result projection includes `boss_scope`, `boss_completion_policy`, Boss start-readiness, registered-player, all-connected/all-ready, and connection-count fields, `boss_friendly_fire_policy`, `boss_max_hp`, `boss_current_hp`, `boss_damage_total`, per-player `boss_damage_<player_id>` contribution fields, per-player `boss_player_<player_id>_spawn_slot`/`fire_target` layout fields, `boss_defeated`, `boss_clear_status`, and `boss_result_disposition`. World Boss matches report `world_damage_report` for business-side persistent HP settlement, while instance Boss matches report `instance_cleared` only after HP reaches zero and `instance_incomplete` otherwise. `SubmitBattleResult` binds the aggregate and per-player Boss fields back to the server-owned final replay snapshot before accepting the signed result. When a Boss `transfer_card` action was accepted, the callback also binds the transfer count, card instance id, from/to player ids, and frozen authority flags to the final snapshot. This is settlement intent material for Nakama/Go verification, not reward or database mutation.
- The development `mode_result_json` payload is built from the canonical replay fixture at the simulation boundary, and the server facade reuses that single payload when creating signed result callbacks. Tests must submit server-built signed-result material and tamper it for negative cases rather than reconstructing callback JSON in test helpers, keeping the protobuf-replacement result boundary tied to the same replay fixture hash source.
- Development replay fixtures are generated only from the server-owned simulation summary and final authoritative snapshot. They include replay id, owner user id, match/mode/ruleset, match seed, ordered player ids, result hash, event cursor, deterministic input/event trace material, final snapshot, 60Hz tick-rate metadata, a manifest-compatible `ReplayInputStreamSummary` record, canonical fixture payload/hash material, and the server-authoritative flag so generated protobuf replay/result bindings can replace the scaffold without changing the tested boundary. The canonical fixture payload now serializes the match seed and final snapshot body, including sorted player positions/connectivity, bullet deltas, and mode-state fields, so a fixture hash changes if replay-visible snapshot material is tampered while aggregate hashes remain unchanged.
- Development replay records are generated through `BuildReplayRecord` as a dependency-light bridge for the future `ReplayRecord` protobuf. The bridge binds replay id, owner user id, match/mode/stage ids, ticket-derived loadout refs with user/player/stage/deck snapshot hash/ruleset material, the manifest-compatible `ReplayInputStreamSummary` stream, the server-built signed battle result settlement, server-authoritative flags, creation time, canonical payload material, and a deterministic development hash. It is export material only; it does not persist replay rows or settle rewards.
- Battle results are bound to the allocated match id, frozen mode id, frozen ruleset version, match seed, player ids, local replay-derived result hash, replay id, replay event cursor, final tick, fixed tick rate, replay input/event/final-state hashes, manifest-compatible replay summary record hash, canonical replay fixture hash, Boss settlement projection fields when present, settled time, battle server key id, and deterministic development signature before the business server can settle rewards.
- Between result acceptance and `RetireMatch`, mutating match entrypoints reject with `match_settled` and `TickMatch` returns a held `match_settled` snapshot, so late packets cannot change signed replay/hash material while read-only audit access remains available.
- Battle result reward JSON is treated as a projection only. Obvious inventory, wallet, grant, balance, database, or Steam inventory mutation fields are rejected at the C++ boundary so Nakama/Go remains the only asset-settlement authority.
- Battle result mode JSON is also treated as replay/audit material only. The result verifier rejects obvious inventory, wallet, currency, grant, item, balance, database, Steam inventory, or client-result-authoritative fields there too, so extra audit keys cannot smuggle business mutation intent through the mode projection channel.
- The battle server never writes inventory, rewards, wallet, or database state. It only emits replay/hash/cursor-bound result material that the business server can verify and settle idempotently.
- Client packet seq numbers must increase per `match_id:player_id`.
- Business session references must be opaque references, not raw bearer tokens.
